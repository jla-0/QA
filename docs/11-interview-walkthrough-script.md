# Interview Walkthrough Script

Use this to explain the portfolio project in an interview without sounding scripted.

## 30-Second Version

I built this healthcare claims QA portfolio to show how I would perform the Claritev QA Manual Tester role. It walks from requirements analysis into test strategy, traceability, manual test cases, SQL backend checks, SOAP/XML and EDI examples, Jira-style defects, HIPAA-safe test data, regression, integration, and load testing planning. It is synthetic, so it is safe to share, but the workflow is realistic.

## 90-Second Version

My professional background combines regulated systems QA, validation evidence, SQL/data analysis, healthcare IT support, and audit-ready documentation. Since this role is specifically healthcare claims, I created a claims-focused QA project to demonstrate the job workflow directly.

The project starts with requirements and risk analysis, then builds a test strategy and traceability matrix. From there I created manual test cases for claim intake, eligibility, provider contract validation, adjudication, denial handling, payment, claim status inquiry, and regression. I also added SQL checks to validate backend data, SOAP/XML samples for claim status, EDI-style 837 and 835 files, Jira defect examples, and a HIPAA test data strategy.

The main idea is that a QA tester should not only click through the front end. They should compare the UI, database, integration responses, and business rules so the team understands the real quality risk.

## Strong Answer To Claims Experience Question

My direct healthcare background is in clinical IT systems, PACS/DICOM workflows, HIPAA-aware support, and regulated documentation. My direct QA background is in validated systems, SQL/data checks, requirements traceability, test execution, and defect-style investigations. Because this role is claims-specific, I built this portfolio walkthrough using synthetic claims data, EDI examples, SOAP/XML checks, SQL validation, and Jira-style defects to show exactly how I would approach claims QA work.

## Strong Answer To SQL Question

I use SQL testing to confirm that what the UI shows is actually persisted correctly. In this project, the SQL checks look for status mismatches, paid claims without payment records, denied claims without denial reasons, paid amounts exceeding allowed amounts, duplicate payment risk, inactive member eligibility, inactive provider contracts, and orphan claim lines. That is the kind of backend validation that catches defects a front-end-only pass can miss.

## Strong Answer To HIPAA Question

For testing, I would default to synthetic data whenever possible. If real data is necessary, I would use only approved environments, minimum necessary access, and redacted evidence. I would avoid putting PHI in screenshots, Jira tickets, chat, local files, or public documentation. This portfolio follows the same discipline: all claim, member, provider, XML, and EDI examples are synthetic.

I may not come from a pure claims-only title, but I can show the working method and the artifacts I would use to contribute quickly.
